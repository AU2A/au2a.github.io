【Temperature 1.5 的日常】EP4: LangChain - Streaming, 打破沈默的呼吸感

Aura
2025-10-17
Temperature 1.5
LangChain, Python

本文為個人學習筆記，記錄了學習過程中的一些知識，可參考，但不可認真，學習的過程可能有理解錯誤，資訊不一定正確，畢竟 Temperature 都 1.5 了。

零、 前言

在上一篇我們討論了 Tools，讓 AI 可以執行動作。但你有沒有發現，當 AI 在調用工具、查資料、思考複雜邏輯時，使用者往往對著一個閃爍的游標發呆？

在 LLM 的時代，「延遲（Latency）」是開發者的天敵。由於模型是逐個 Token 產生的，如果等到 500 個 Token 全部生完才顯示，使用者可能已經以為網頁當機了。這就是為什麼我們需要 **Streaming (串流)**。這不只是為了視覺效果，更是為了提升「感知速度」。

一、 串流的底層邏輯：逐字吐露

為什麼 LLM 可以串流？因為自回歸模型（Autoregressive Models）本質上就是根據前文預測下一個字。

當模型算出 時，我們其實不需要等它算出 ，就可以先把第一個字送到前端。

二、 LangChain 的三種串流姿勢

在 LangChain 中，串流的實作經歷了幾次演進，現在主要有以下幾種方式：

1. 最基礎的 .stream()

這是同步（Sync）環境下最簡單的方式，它會回傳一個 Generator（生成器）。

2. 非同步的 .astream()

在現代的 Web 框架（如 FastAPI）中，我們通常使用非同步方式來處理併發請求。

3. 終極方案：.astream_events()

這是目前 LangChain 最強大也最推薦的方式。為什麼？因為 .stream() 只能拿到最終的文字，但如果你使用了 Chain 或 Agent，你可能想知道：

  • 現在是不是正在呼叫工具？
  • 哪一個步驟正在檢索資料庫？
  • 中間層的思考過程是什麼？

astream_events 允許你訂閱整個鏈路中的所有事件。

三、 實戰範例：監聽 Agent 的呼吸

想像一個情境：AI 正在幫你搜尋資料並整理報告。我們希望前端能即時顯示：「正在搜尋…」、「找到 3 篇文章…」、「正在總結…」。

四、 串流中的常見挑戰

雖然串流看起來很美好，但實作時有幾個「坑」：

  1. JSON 格式崩潰：如果你在要求模型輸出 Structured Output（JSON），串流出來的片段在還沒傳完之前，都不是合法的 JSON。這會導致前端 JSON.parse() 報錯。
  • _對策_：使用 Partial JSON Parser 或是在後端聚合完整後再輸出。
  1. **聚合 (Aggregating Chunks)**：LangChain 的 AIMessageChunk 支持相加。

這在背後會自動幫你處理 tool_calls 的合併，非常方便。 3. Token 計費：在串流模式下，有些 Provider 可能不會回傳完整的 Token 使用統計。

五、 前端如何接收？（SSE 協定）

串流數據通常不建議用純 WebSocket（除非有雙向通訊需求），最標準的做法是使用 **Server-Sent Events (SSE)**。

特性 REST API (JSON) SSE (Streaming)
等待感 高 (等到全部結束) 低 (即時回饋)
連線方式 短連線 長連線
資料格式 單一物件 數據流 (text/event-stream)

六、 結語

串流不只是技術上的實現，更是一種 UX 哲學。在 AI 回應緩慢的現實下，透過逐字顯示，我們給予了使用者一種「AI 正在努力思考並與你對話」的臨場感。

當你學會了 Tools（手）與 Streaming（呼吸），你的 AI Agent 就不再是一個死板的 API，而是一個鮮活的、有節奏感的對話夥伴。
